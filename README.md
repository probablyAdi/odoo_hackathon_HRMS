# HRMS вЂ” Human Resource Management System

*Every workday, perfectly aligned.*

A full-stack HRMS built for the hackathon: auth, role-based dashboards, employee
profiles, attendance, leave/time-off, and payroll вЂ” built on **FastAPI +
PostgreSQL** with a **plain HTML/CSS/JS** frontend (no framework, no build step).

## Stack

| Layer      | Choice                                                        |
|------------|----------------------------------------------------------------|
| Backend    | Python, FastAPI, SQLAlchemy ORM                                |
| Database   | PostgreSQL                                                      |
| Auth       | JWT (python-jose) + bcrypt password hashing (passlib)          |
| Frontend   | Plain HTML / CSS / vanilla JS (no React, no build tooling)     |
| Data       | 100% real, dynamic data from Postgres вЂ” no Firebase, no mocked JSON in the final app |

## Getting started

### 1. Database

```bash
createdb hrms
psql -d hrms -f backend/schema.sql
```

(SQLAlchemy will also auto-create any missing tables on first run, but running
`schema.sql` yourself is the recommended path since it's the reviewable source
of truth.)

### 2. Backend

```bash
cd backend
python -m venv venv && source venv/bin/activate      # Windows: venv\Scripts\activate
pip install -r requirements.txt
cp .env.example .env                                  # then edit DATABASE_URL / JWT_SECRET_KEY
python seed.py                                         # optional: demo data
uvicorn app.main:app --reload
```

API docs (auto-generated by FastAPI): http://localhost:8000/docs

### 3. Frontend

No build step вЂ” just serve the static folder:

```bash
cd frontend
python -m http.server 5500
```

Open http://localhost:5500. If you seeded demo data, sign in with
`priya.admin@example.com` / `Passw0rd!` (admin) or `john.doe@example.com` /
`Passw0rd!` (employee).

If your backend runs anywhere other than `http://localhost:8000`, update
`API_BASE` at the top of `frontend/js/api.js`.

## How the spec maps to the build

- **Login ID auto-generation** (`[Company Code][First2][Last2][Year][Serial]`,
  e.g. `OIJODO20260001`) вЂ” `backend/app/utils/id_generator.py`.
- **Admin creates employees, not self-registration** вЂ” only the *first* admin
  of a company signs up publicly (`POST /api/auth/signup`); every subsequent
  employee is created by an admin (`POST /api/employees`), which generates a
  login ID + temporary password and forces a password change on first login.
- **Role-based access** вЂ” `require_admin` dependency guards every admin-only
  route; employees can only ever edit their own limited fields.
- **Salary auto-calculation** вЂ” admin sets the wage and a handful of %
  rules; every component amount is derived server-side
  (`utils/salary_calc.py`), so the "components never exceed the wage" rule
  holds by construction (Fixed Allowance is the remainder).
- **Attendance status dot** (red в†’ green on check-in) and check-in/out вЂ”
  `attendance.py` + the `status-dot` element wired in `shell.js`.
- **Leave balances, overlap checks, admin approve/reject** вЂ” `timeoff.py`.

## Design & engineering notes

- **Input validation**: Pydantic models validate every request server-side
  (regex-constrained enums, date-range checks, percentage caps); the frontend
  also validates before submitting so errors surface immediately, not just
  after a round-trip.
- **Graceful errors**: a global FastAPI exception handler turns validation
  errors into a flat, frontend-friendly list; integrity errors (e.g.
  duplicate email) return a clear 409 message instead of a stack trace.
  The frontend never shows a raw error object вЂ” always a human sentence.
- **Security**: bcrypt password hashing, JWT bearer auth, generic
  "incorrect login or password" message (no user enumeration), role checks
  on every mutating endpoint, and company-scoped queries so one company's
  admin can never see another company's data.
- **No third-party services**: no Firebase, no external auth or storage
  provider вЂ” Postgres is the only dependency, matching the brief.

## Suggested git workflow for the hackathon

```bash
git init
git add .
git commit -m "Initial scaffold: schema, auth, employee/attendance/leave/payroll APIs"
```

From there, branch per feature (`feature/attendance-ui`,
`feature/payroll-calc`, etc.), open small PRs even solo вЂ” it gives you a
clean commit history to walk judges through, and makes merge conflicts
between teammates much easier to resolve.

## Known simplifications (documented, not hidden)

- Email verification is modeled (`is_email_verified`) but no email is
  actually sent вЂ” fine for a prototype; swap in a transactional email
  provider later if needed.
- Leave day counts are calendar days inclusive of both ends (not
  business-day-aware); see the comment in `timeoff.py` for where to plug in
  a working-calendar lookup if you want weekends excluded.
- A newly created admin employee's temporary password is returned directly
  in the API response so the demo can show it on-screen; in production this
  would be emailed instead, never returned over the API.
